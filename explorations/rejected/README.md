# Rejected Explorations

This folder holds **rejected** technology and design explorations: options I evaluated and explicitly decided against, with reasons documented.

## Purpose

- **Preserve context:** Why was an option rejected?
- **Avoid repeat work:** Prevents re-running the same exploration.
- **Onboarding:** New readers can see past decisions and rationale.
- **Review:** Rejected items can be revisited periodically (e.g. every 6–12 months) to see if circumstances have changed.

## Structure

- **technology/** — Rejected technology choices (e.g. stack, protocols, tooling).
- **game-design/** — Rejected game-design ideas (if any).
- Other categories (e.g. monetization, economy, community) may be added as needed.

## File format

- **Naming:** `YYYY-MM-short-description.md`
- **Content:** Each document should describe the idea, what was considered, and why it was rejected (see the template in the main project repository if you use one). Outcome and “what I use instead” should be clear.

## Important

- **Rejected** does not mean “bad” — it means “not a fit for this project at this time.”
- Decisions are project-specific; other projects may reasonably choose differently.
- If constraints change, a rejected exploration can be reopened.
